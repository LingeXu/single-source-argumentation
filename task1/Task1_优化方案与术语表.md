## 一、优化方案（由易到难）

### P0 — 本周可做，低投入高回报

#### 1. 参数网格搜索脚本

**问题**：当前阈值 0.65/0.85 是拍脑袋定的。50% 的错误来自假 IDK——阈值太紧，把 VLM 的正确回答强行改成了 IDK。

**方案**：写一个轻量脚本 `tune_params.py`，在 50 条验证集上自动测试 20-30 组参数组合：

```
STRICT=0.55, LOOSE=0.75  →  Acc=?, Hallu=?
STRICT=0.55, LOOSE=0.80  →  Acc=?, Hallu=?
...
STRICT=0.70, LOOSE=0.90  →  Acc=?, Hallu=?

最优组合 → 写入 task1_agent.py
```

| 投入 | 收益 |
|------|------|
| 半天 | 假 IDK 减少 20-30% |

---

#### 2. 修复垃圾判定误杀

**问题**：`len(answer) < 3` 一刀切，将 "5""$26,000" 等数字类短答案误判为垃圾。

**方案**：纯数字或含单位的短答案不杀。

| 投入 | 收益 |
|------|------|
| 5 分钟 | 避免正常短答案被误杀 |

---

#### 3. 升级模型 2B → 7B

**问题**：2B 参数模型阅读理解能力太弱，常看漏 KG 数据中的有效信息。

**方案**：将 `task1_agent.py` 中模型改为：

```python
VLM_MODEL_NAME = "mlx-community/Qwen2-VL-7B-Instruct-4bit"
```

首次下载约 5GB，需联网。

| 投入 | 收益 |
|------|------|
| 改一行代码 + 下载 | 准确率 +8~12% |

---

### P1 — 下一周，中等投入

#### 4. 文本提取排序借用规则映射表

**问题**：Step 3 文本提取排序时，"cost" 和 "msrp" 零词重叠 → 正确答案排在最末尾，VLM 注意不到。

**方案**：排序时查映射表，属性命中映射关系 → 加分。

| 投入 | 收益 |
|------|------|
| 加 5 行代码 | VLM 更容易看到关键属性 |

---

#### 5. 嵌入语义查询分类

**问题**：关键词规则太死——不认识 "how much does it cost" ≈ "what's the price"。

**方案**：用 `all-MiniLM-L6-v2`（评估框架自带，无需额外安装）将问题文本转为向量，与四种类型的代表向量做相似度匹配。纯视觉类问题可跳过 KG 检索。

| 投入 | 收益 |
|------|------|
| 半天 | 分类精度 +10-15%，误分类不再影响阈值选择 |

---

#### 6. 动态阈值替代固定阈值

**问题**：固定 0.65/0.85 不适应所有图片。某些图片 CLIP 整体匹配分数偏低，固定阈值导致全部过滤。

**方案**：根据本轮 KG 召回结果的分数中位数动态计算阈值。

| 投入 | 收益 |
|------|------|
| 半天 | 适应不同图片的匹配特性，减少过度过滤 |

---

### P2 — 后续，较大投入

#### 7. 答案可追溯验证

**问题**：VLM 有时编造 KG 中完全不存在的信息，但置信度评估未发现——因为 KG 确实有匹配，只是模型没有认真读。

**方案**：VLM 输出后，搜索答案关键词是否出现在 KG 属性值中。知识型问题若答案无源文件支撑 → 标记为可疑，改 IDK。

| 投入 | 收益 |
|------|------|
| 1 天 | 幻觉率 -5~8% |

---

#### 8. Re-ranker 精排

**问题**：CLIP 召回 10 条 KG 实体，排序可能不精，Top-5 中夹杂无关实体。

**方案**：粗召回后，使用 `BGE-reranker-v2-m3` 对（问题，实体描述）对做交叉编码精排，从中选出真正相关的 Top-5。

| 投入 | 收益 |
|------|------|
| 2-3 天 | 检索精度 +10~15% |

---

#### 9. 闭环错误分析

**问题**：每次评估只知道分数，不知道具体哪类错误最多、优化方向何在。

**方案**：评估后自动输出错误分类报告——KG 漏配占比、VLM 事实错误占比、过度保守占比——下一轮优化直接瞄准头号问题。

| 投入 | 收益 |
|------|------|
| 1 天 | 持续迭代的效率倍增器 |

## 二、专有名词解释

### 核心概念

| 名词 | 全称 | 通俗解释 |
|------|------|---------|
| **RAG** | Retrieval-Augmented Generation | 检索增强生成。不让大模型凭空编答案，先让它"翻书查资料"，再根据资料回答。类比：开卷考试 vs 闭卷考试 |
| **KG** | Knowledge Graph | 知识图谱。结构化的知识库，数据像表格一样存着——"实体：丰田凯美瑞 → 座位数：5 → 品牌：丰田"。比网页搜索更可靠 |
| **VLM** | Vision-Language Model | 视觉语言模型。能同时"看图片"和"读文字"的大模型。普通 ChatGPT 只能读字，VLM 还能看图 |
| **MM-RAG** | Multi-Modal RAG | 多模态检索增强生成。RAG 的升级版——检索的不仅是文字，还包括图片、表格等不同形态的信息 |

### 模型与嵌入

| 名词 | 全称 | 通俗解释 |
|------|------|---------|
| **CLIP** | Contrastive Language-Image Pre-training | OpenAI 出的模型。能把图片和文字映射到同一个数学空间，计算它们"有多像"。Task 1 用它把用户图片和 KG 图片做匹配 |
| **BGE** | BAAI General Embedding | 智源研究院出的文本嵌入模型。把文字变成数学向量，算两段话语义有多相关。Task 2 用它做 Web 搜索 |
| **Embedding** | 向量嵌入 | 把图片/文字转换成固定长度的一串数字（向量），让计算机能计算相似度 |
| **Re-ranker** | 重排序模型 | 在粗检索之后，用更强的模型对候选结果做精细重排。类比：先海选（CLIP），再复赛（Re-ranker），最后冠军（VLM 用） |

### 任务与评估

| 名词 | 全称 | 通俗解释 |
|------|------|---------|
| **Hallucination** | 幻觉 | 大模型瞎编——检索资料里没有这个信息，它自己编了一个。这是赛题最想惩罚的行为 |
| **Missing** | 缺失 | 系统主动说 "I don't know"。比赛中不算错，比瞎编好 |
| **Accuracy** | 准确率 | 回答语义正确的比例 |
| **Truthfulness Score** | 真实性得分 | 赛题最终排名依据。正确得 +1，IDK 得 0，错误得 -1 |
| **Exact Match** | 精确匹配 | 回答和标准答案字符串完全一致才算对。过于严格，通常需要语义评估补充 |

### 系统设计

| 名词 | 通俗解释 |
|------|---------|
| **Prompt** | 发给大模型的指令文本。分 System Prompt（告诉模型怎么做事）和 User Prompt（具体任务内容）。Prompt 怎么写直接决定模型输出质量 |
| **Few-shot** | 在 Prompt 里塞几道例题+标准答案，让模型"看例题学答题"，而不是口头描述规则。代码里给了 5 个例题教模型：有数据→取值、没数据→看图、无关→说 IDK |
| **Temperature** | 温度参数。控制模型输出有多"随机"。0.0 = 每次一样（适合问答），1.0 = 每次不同（适合创意写作）。Task 1 设为 0.0 |
| **System Prompt** | Prompt 中"教规矩"的部分。告诉模型它是什么角色、应该怎么回答。Task 1 的 System Prompt 包含 Few-shot 示例和行为规则 |
| **User Prompt** | Prompt 中"给任务"的部分。包含 KG 数据和用户具体问题，是模型真正要处理的内容 |
| **CLIP 余弦距离** | 两张图片的相似度。范围 0~1，0=一模一样，1=完全不相关。代码里知识型问题用 0.65 阈值，视觉型用 0.85 |

### Apple Silicon 相关

| 名词 | 全称 | 通俗解释 |
|------|------|---------|
| **MLX** | — | Apple 出的机器学习框架，专门为 Mac 的 M 系列芯片优化。类比：CUDA 是 NVIDIA 显卡的加速器，MLX 就是 Mac 芯片的加速器 |
| **4-bit 量化** | — | 把模型参数从 16 位压缩到 4 位存储。模型大小减少约 4 倍，轻微损失精度，换来能在普通电脑上运行 |

## 三、Task 1 流程速查

```
图片 📸 + 问题 ❓
    │
    ├─ Step 1: 查询分类         → knowledge / visual / reasoning / comparison
    ├─ Step 2: KG 检索+过滤      → CLIP 匹配 + 双阈值过滤 (0.65 / 0.85)
    ├─ Step 3: 信息提取
    │   ├─ 路 A: 规则提取       → 确定性取值（窄而准，兜底用）
    │   └─ 路 B: 文本提取       → 洗数据给 VLM 读（宽而全，主力用）
    ├─ Step 4: Prompt 构建       → System + Few-shot 例题 + KG 文本 + 问题
    ├─ Step 5: VLM 生成          → Qwen2-VL-2B，温度 0.0
    ├─ Step 6: 后处理            → 去前缀 / 截断 / 统一 IDK
    └─ Step 7: 置信度评估        → KG 无匹配→强制 IDK；VLM 失效→路 A 兜底
                    ↓
                最终答案
```

### 关键参数

| 参数 | 默认值 | 位置 | 说明 |
|------|--------|------|------|
| `VLM_MODEL_NAME` | Qwen2-VL-2B-4bit | task1_agent.py | 改为 7B 可提升准确率 |
| `NUM_KG_RECALL` | 10 | task1_agent.py | KG 召回数量 |
| `NUM_KG_FINAL` | 5 | task1_agent.py | 过滤后保留数量 |
| `STRICT_THRESHOLD` | 0.65 | task1_agent.py | 知识型问题过滤阈值 |
| `LOOSE_THRESHOLD` | 0.85 | task1_agent.py | 视觉型问题过滤阈值 |
| `TEMPERATURE` | 0.0 | task1_agent.py | 生成温度 |
| `MAX_GENERATION_TOKENS` | 128 | task1_agent.py | 最大生成长度 |
| `BATCH_SIZE` | 4 | task1_agent.py | 批处理大小 |

### 错误分布

| 错误类型 | 占比 | 根因 |
|---------|:---:|------|
| 过度保守（假 IDK） | ~50% | 阈值太严，VLM 正确回答被强制改为 IDK |
| KG 检索漏配 | ~20% | CLIP 未能匹配到正确 KG 实体 |
| VLM 事实错误 | ~30% | 2B 模型推理/提取能力不足 |
